# Phase 9c — Tax + 1099 + Reconciliation Specification

> **Author note (transparency):** This spec is being written ahead of when Phase 9c will actually be built. The spec is correct as of the current date but will need a refresh when implementation actually starts, particularly for: actual cleaner YTD distribution, Stripe Connect 1099 flag mechanics (subject to API changes), and reconciliation discrepancy patterns observed at scale. Treat this as an aggressive draft.

**Phase goal:** Year-to-date earnings tracked per cleaner (cached daily). $600 IRS threshold flagged in Stripe Connect; 1099-NEC generated by Stripe at year-end. Admin financial dashboard renders platform revenue trends with time-series + CSV export. Year-end triple-entry reconciliation report ties out customer payments + tips - cleaner payouts - refunds - platform fees = $0 (within rounding tolerance).

**Estimated duration:** ~1.5 weeks of focused engineering (8 working days).

**Depends on:**
- Phase 9a complete (ledger + revenue events flowing)
- Phase 9b complete (payouts populating)
- Stripe Connect 1099 generation feature enabled
- Phase 4 cleaner tax info captured (W-9 / W-8BEN equivalent on Stripe)

**Wireframes covered:** WF 30 (1099 status, partial — Stripe handles bulk of UI), WF 34 (tax info form already specced in Phase 4), admin financial dashboard (no specific wireframe — scratch design).

**Phase 9c sub-sections (mostly parallel within 9c):**

- **9c-1** — Year-to-date earnings tracker (~2 days)
- **9c-2** — Admin financial dashboard (~3 days)
- **9c-3** — Year-end reconciliation report (~2 days)
- **Closeout + integration** (~1 day)

## 1. Summary

Phase 9c is **the platform's tax + financial reconciliation layer.** Concretely, by the end of Phase 9c:

1. **YTD earnings tracked per cleaner.** Daily cron sums earning events from `cleaner_ledger_events`. Cached on `cleaner_profiles.ytd_earnings_cents`.

2. **$600 IRS threshold flagged in Stripe.** When cleaner crosses, Stripe flagged for 1099 generation at year-end.

3. **WF 30 surfaces 1099 status.** "On track for 1099 / Crossed $600 / 1099 issued."

4. **Admin financial dashboard renders platform metrics.** Daily/weekly/monthly GMV + revenue + refund rate + goodwill + payout totals + top earners.

5. **Time-series visualizations.** 14d / 30d / 90d / YTD views.

6. **CSV export.** Admin downloads for accounting team.

7. **Year-end reconciliation report.** Triple-entry: customer + cleaner + platform sides balance to $0 (within rounding). Discrepancies surfaced for investigation.

What Phase 9c does NOT do:
- Direct 1099 generation (Stripe handles)
- Tax filing for cleaners (their responsibility)
- State tax variations (federal only v1)

## 2. Acceptance criteria

### 9c-1 YTD earnings tracker

- [ ] Daily cron at 5 AM Pacific (after Friday payout)
- [ ] Computes per cleaner: `SUM(cleaner_ledger_events.amount_cents WHERE created_at YEAR = current AND event_type IN ('charge_captured_cleaner_share', 'tip_received'))`
- [ ] Refunds (`refund_*_clawback`) reduce earnings
- [ ] Payouts (`payout_sent`) don't affect YTD (balance movements, not earnings)
- [ ] Cache on `cleaner_profiles.ytd_earnings_cents`
- [ ] When YTD crosses $600: Stripe Connect flagged via API
- [ ] WF 30 displays cleaner's YTD earnings + 1099 status
- [ ] Year boundary: Jan 1 0:00 Pacific resets YTD to 0

### 9c-2 Admin financial dashboard

- [ ] `/admin/financial-dashboard` route accessible only to admin
- [ ] Reads from `platform_revenue_events` + aggregations
- [ ] Key metrics rendered:
  - Daily/weekly/monthly GMV (sum of customer payments)
  - Daily/weekly/monthly platform revenue (booking fees + commissions - goodwill - refund returns)
  - Refund rate (refunds / completed bookings)
  - Goodwill absorption total
  - Cleaner payout totals
  - Top 10 earners + bottom 10 performers
- [ ] Time-series toggle: 14d / 30d / 90d / YTD
- [ ] Charts render <500ms p95
- [ ] CSV export: all events for selected date range; downloads <5 seconds for 90 days
- [ ] Daily aggregation `platform_revenue_daily` cache populated by cron

### 9c-3 Year-end reconciliation

- [ ] Annual cron January 31 of following year runs reconciliation
- [ ] Triple-entry calculation:
  - Customer side: SUM(charges.amount) + SUM(tips.amount) - SUM(refunds.amount) = X
  - Cleaner side: SUM(cleaner_ledger_events) = Y
  - Platform side: SUM(platform_revenue_events) = Z
  - Identity: X = Y + Z (within ±$1 rounding tolerance)
- [ ] Discrepancies > $100 flagged for admin investigation
- [ ] Stripe reconciliation: compare to Stripe's monthly reports
- [ ] Report PDF generated + delivered to admin
- [ ] Discrepancy review interface for admin

### Cross-cutting

- [ ] All Phase 9c code has unit tests; coverage ≥80%
- [ ] RLS: admin reads platform revenue + reconciliation; cleaner reads own YTD
- [ ] Performance: dashboard load <500ms p95
- [ ] CSV export memory-efficient (streaming, not in-memory)

### New migrations (Phase 9c)

```sql
-- Phase 9c migration

-- Reconciliation reports tracking
CREATE TABLE reconciliation_reports (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  report_year INTEGER NOT NULL,
  customer_side_cents BIGINT NOT NULL,
  cleaner_side_cents BIGINT NOT NULL,
  platform_side_cents BIGINT NOT NULL,
  discrepancy_cents BIGINT NOT NULL,
  status TEXT NOT NULL DEFAULT 'computed' CHECK (status IN ('computed', 'reviewed', 'reconciled')),
  generated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  reviewed_by_admin_id UUID REFERENCES users(id) ON DELETE SET NULL,
  reviewed_at TIMESTAMPTZ,
  notes TEXT,
  UNIQUE (report_year)
);

-- Discrepancy investigations
CREATE TABLE reconciliation_discrepancies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  report_id UUID NOT NULL REFERENCES reconciliation_reports(id) ON DELETE CASCADE,
  side TEXT NOT NULL CHECK (side IN ('customer', 'cleaner', 'platform')),
  amount_cents BIGINT NOT NULL,
  description TEXT,
  resolved BOOLEAN NOT NULL DEFAULT FALSE,
  resolution_notes TEXT,
  resolved_at TIMESTAMPTZ,
  resolved_by_admin_id UUID REFERENCES users(id) ON DELETE SET NULL
);
CREATE INDEX idx_discrepancies_report ON reconciliation_discrepancies (report_id);

-- 1099 status cache on cleaner_profiles
ALTER TABLE cleaner_profiles
  ADD COLUMN IF NOT EXISTS irs_1099_threshold_crossed_at TIMESTAMPTZ;
ALTER TABLE cleaner_profiles
  ADD COLUMN IF NOT EXISTS irs_1099_issued_at TIMESTAMPTZ;
ALTER TABLE cleaner_profiles
  ADD COLUMN IF NOT EXISTS irs_1099_year INTEGER;
```

### RLS policies

```sql
-- Reconciliation: admin only
ALTER TABLE reconciliation_reports ENABLE ROW LEVEL SECURITY;
CREATE POLICY reconciliation_admin_only ON reconciliation_reports
  FOR ALL USING (
    EXISTS (SELECT 1 FROM admin_users WHERE user_id = auth.uid())
  );

ALTER TABLE reconciliation_discrepancies ENABLE ROW LEVEL SECURITY;
CREATE POLICY discrepancies_admin_only ON reconciliation_discrepancies
  FOR ALL USING (
    EXISTS (SELECT 1 FROM admin_users WHERE user_id = auth.uid())
  );
```

## 6. Specific gotchas

### Gotcha A — Year boundary timezone

**The problem:** Cron runs UTC. Year boundary is Jan 1 0:00 Pacific = Jan 1 8:00 UTC. Cleaner earns at Jan 1 7:00 Pacific → events count toward old year YTD.

**The fix:** YTD calc uses Pacific year, not UTC year. `EXTRACT(YEAR FROM created_at AT TIME ZONE 'America/Los_Angeles')`. Document.

### Gotcha B — Stripe Connect 1099 flag timing

**The problem:** Cleaner crosses $600 March 15. Stripe flagged. Cleaner has refund Apr 1, brings YTD back below $600. Stripe still flagged.

**The fix:** Stripe 1099 flag is sticky once set. Once crossed, 1099 generates regardless of subsequent dips. IRS rule: total earnings, not net at year-end. Document.

### Gotcha C — Reconciliation rounding

**The problem:** Triple-entry math involves cents arithmetic with refund partial percentages. Cumulative rounding causes $5 discrepancies.

**The fix:** ±$1 rounding tolerance per report acceptable. Discrepancies >$100 flagged for investigation. Stripe is source of truth for any single transaction.

### Gotcha D — Stripe vs PureTask reconciliation drift

**The problem:** PureTask records charge $125.99. Stripe records $125.99 minus their fee 2.9% + $0.30 = $122.04. Difference is Stripe's fee, not a discrepancy.

**The fix:** Reconciliation accounts for Stripe processing fees as separate line item. Don't try to make platform_revenue match Stripe net — they differ by Stripe's fee.

### Gotcha E — Year-end ratio at scale

**The problem:** 10,000 cleaners × 1099 generation in Stripe = bulk delivery January 31. Stripe handles. PureTask doesn't.

**The fix:** Trust Stripe Connect to handle bulk 1099 generation. Phase 9c just provides the YTD signal. If Stripe issues delays, customer service path: "Contact Stripe support; we use Stripe Connect for 1099s."

### Gotcha F — Cleaner disputes 1099 amount

**The problem:** Cleaner sees $14,820 on 1099. Their records show $14,950. Calls support.

**The fix:** Phase 9c YTD calc + Stripe 1099 should match (both pull from cleaner_ledger_events / Stripe's ledger). Discrepancies indicate platform bug. Investigate per-event level.

### Gotcha G — Dashboard query slow at scale

**The problem:** 1 year of platform_revenue_events × 1000 cleaners = millions of rows. Dashboard query slow.

**The fix:** Daily aggregation cache (`platform_revenue_daily`). Dashboard queries cache, not raw events. Cache freshness: 24 hours; admin sees yesterday's totals on first morning load.

### Gotcha H — CSV export memory

**The problem:** 90 days of events = potentially 100K rows. Loading all into memory crashes server.

**The fix:** Streaming CSV: write rows as queried. `cursor` pagination. Memory stays low regardless of export size.
